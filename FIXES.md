CRUCIAL PERFORMANCE
Every frame, paint() calls this.readTokens() 15 or more times across sub-methods. Each call does a null check and property dereference. At 60fps with 3 equations this is over 900 unnecessary function calls per second. Fix: assign const tokens = this.readTokens() once at the top of paint() and pass it as a parameter to every private draw method that currently calls readTokens() internally.
sampleEquation is called every single frame inside drawCurve for every visible equation with no frame-to-frame geometry cache. This is the single largest render CPU bottleneck. Add a per-equation curve segment cache inside CanvasRenderer keyed by equation.id + equation.raw + viewportBucket + variablesHash. On a cache hit, skip sampleEquation entirely and use the cached Float64Array segments. Invalidate only when any of those four keys change. This alone should cut render CPU by 60–80% during normal panning which changes the viewport bucket but not the math.
All cache key constructions in the renderer currently call Object.values(this.state.variableScope()).join(':') inline. This builds a new string every frame even when variables have not changed. Compute a stable variablesHash string once per variable scope change inside the variableScope() method and expose it as a separate property, then reference it in all cache key constructions. The variableScope() memoization already detects changes — just extend it to also update a hash string.
clipLineToRect returns a new 4-element tuple array on every call. With 2400 points per equation at 60fps with 3 equations, this allocates 432,000 short-lived arrays per second. Change the function to accept a pre-allocated output array parameter [number, number, number, number] and write into index 0–3 rather than returning a new allocation. Callers pass a reusable buffer.
Inside adaptiveSample, refineCartesian creates {...scope, x: midpointX} on every recursive call at depths up to 8. Pre-allocate a single mutable scope object outside the recursion, set scope.x directly before each call, and restore it after. This eliminates all object allocation in the refinement hot path.
The MemoryBoundCanvasCache eviction loop runs synchronously on every cache.set() call. During rapid pan/zoom, polar curves and scatter data trigger many cache.set() calls per rAF, each running the while-eviction loop. Batch eviction: track a dirty flag and run eviction once per rAF cycle in a separate microtask rather than synchronously on every set.
The canvas renderer's viewportBucket() is computed once per paint and passed down — good. But it is still recomputed fresh on every render call including during animation frames where the viewport hasn't changed (e.g. trace-point updates). Cache the last viewport state and only recompute the bucket string when originX, originY, scaleX, or scaleY actually change.
The analysis worker re-parses equations on every criticalPoints, intersections, fitBounds, and equationAnalysis request independently. Add a worker-side LRU parse cache keyed by raw+kind (max 50 entries) using a simple Map with manual eviction. A single parseEquation() call with a complex expression takes 2–5ms. Eliminating re-parsing across repeated viewport-change requests saves tens of milliseconds per pan.
The fitAll sampling is now correctly offloaded to the analysis worker. But the worker receives equation payloads and re-parses raw expressions on every fitBounds message. The worker-side parse cache above fixes this. Verify the worker parse cache is shared across all message types (criticalPoints, intersections, fitBounds, equationAnalysis) not just one.
The KaTeX render cache is a per-module LruMap(200) in both EquationCard and AnalysisPanel, meaning the same equation KaTeX is cached twice in two separate maps. Extract the cache to a singleton module $lib/utils/katex-cache.ts that exports a single shared LruMap(300) and import it in both components.
The ExpressionEditor mathLinter calls parseEquation on every CodeMirror update via the linter debounce. But updateEquation in graph.svelte.ts already calls parseEquation through parseEquationState and stores the error in equation.errorMessage. Pass the current equation's compiled and errorMessage as props to ExpressionEditor and derive the linter diagnostics from those props (just position the underline using errorMessage presence) rather than re-parsing in CodeMirror. This eliminates the duplicate parse on every keystroke.
The DataPanel table's every cell blur calls updateDataSeries which calls graph.dataSeries.splice() which triggers bumpRenderVersion(false) which calls requestRender(). This causes a canvas repaint on every cell edit even when the series is not plotted. Guard bumpRenderVersion in updateDataSeries behind a check: if (!current.plotted && !patch.plotted) skip the render bump.
The VariableSliderPanel calls graph.updateVariable on every oninput event with no rate limiting. During variable animation via toggleAnimation, the rAF loop calls updateVariable up to 60 times per second. Each call triggers clearAnalysisCaches() inside bumpRenderVersion(true) which destroys all cached critical points and intersections, flooding the analysis worker with new requests every frame. Apply a 100ms debounce on the analysis cache invalidation only (not the render) by separating bumpRenderVersion into two calls: requestRender() immediately for visual update, and scheduleAnalysisInvalidation() with debounce for cache clearing.
The ColorPicker adds a document-level pointerdown event listener at component instantiation time, not inside onMount. With 10 equation cards, 10 separate document listeners are registered. Replace with a singleton listener manager: a module-level Set<ColorPicker> of open pickers, one shared document listener that iterates the Set, and components register/unregister themselves. This reduces to a single document listener regardless of how many pickers exist.
The formatAxisLabel function calls new Intl.NumberFormat('en-US', { maximumFractionDigits: 0 }).format(v) for large numbers on every axis label draw, potentially 20+ times per frame. Create a module-level cached Intl.NumberFormat instance and reuse it.

Verify that all callable mathjs functions outside FUNCTION_TOKENS are unreachable. Specifically test: math.import(), math.config(), math.createUnit(), and math.derivative() as FunctionNode targets. The importJSON function should yield between equation parses using setTimeout(0) or a microtask queue when importing more than 10 equations, to avoid blocking the main thread for hundreds of milliseconds with complex expressions.

The localStorage session save fires in a $effect that depends on graph.historyIndex and graph.historySize, triggering on every history commit (every 180ms during editing). FUNCTIONALITIES TO IMPROVE
The polar renderer should detect curve closure by testing r(θ) ≈ r(θ + 2π) before starting the full sample loop. If the equation has period 2π, stop after one full cycle instead of always sampling to 4π. This prevents double-sampling closed curves.
The trace mode currently only traces Cartesian equations. Extend to parametric: find the nearest point on the parametric curve (by testing a sampled set of t values) and display (x, y, t=value) in the trace chip.
The critical point label positioning on canvas does not handle overlap. When two critical points are close together their labels overlap. After computing all marker canvas positions, run a greedy overlap resolution pass: for any two labels whose bounding boxes overlap, offset the lower-priority one vertically by labelHeight + 4px and draw a 1px leader line from the offset label to the marker.
The analysis worker should post partial results rather than waiting for the complete analysis. The analysis worker currently computes all sections (criticalPoints, domain, range, period, symmetry, asymptotes, derivative, integral) in sequence before posting the result. Restructure the equationAnalysis worker handler to post an intermediate result after criticalPoints are computed, then continue and post the final complete result. The analysis panel must handle partial updates without flickering.
The undo history should be bounded not just by entry count (MAX_HISTORY_ENTRIES=50) but also by total size. Add a running totalBytes counter (rough estimate: snapshot JSON length _ 2 for UTF-16). When totalBytes > 10MB, remove the oldest entry regardless of count. This prevents memory issues in very long sessions.
The equation kind change from Cartesian to Parametric currently discards the user's expression without warning and replaces it with the default parametric template. Before changing kind, check if the current raw expression is non-empty and non-default. If so, commit the current raw to history and offer an undo path. Show a brief toast "Equation type changed — undo to revert."
